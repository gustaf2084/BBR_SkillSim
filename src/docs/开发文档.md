## 一、项目概述

### 1.1 软件定位

一款面向《战场兄弟》重铸 mod 硬核玩家的 Windows 桌面工具软件。重铸 mod 引入「动态特技树生成系统」——角色技能树由背景、特性、天赋、初始武器按**隐藏权重加权随机**生成，游戏内不展示任何权重数据。本软件把这些隐藏规则**可视化、可计算**，把玩家从「靠运气与经验」升级为「靠数据决策」，解决三类痛点：

- **招人盲目**：花高价招募却不知道能拿到什么技能树 → 正向模拟解决（招前预判）
- **选人无据**：有目标 build 却不知道该招什么背景/特性 → 反向推导解决（从需求倒推选人，最实用）
- **上手门槛高**：不知道某背景该怎么培养 → 流派推荐解决

### 1.2 三大功能

1. **正向模拟**：选背景 + 特性，计算所有技能树组（Perk Group）的**单组出现概率分布**，并以 7 阶技能树矩阵可视化；无合法方案时提示「无可用方案」并给出原因。
2. **反向推导**：选目标技能树组（支持多组），列出使目标组出现概率最大的全部「背景+特性」组合，按概率得分与纯度排序。
3. **流派推荐**：自动数据分析三卡片（防御/攻击/特殊）+ 用户自定义流派文件（`builds/*.txt`，可新建/编辑/分享）+ 技能树矩阵。

### 1.3 技术栈

- Python 3.10+
- PySide6（Qt for Python）GUI 框架
- PyInstaller 单文件 exe 打包（`--onefile`，console=False）
- 无网络依赖的纯本地应用
- 代码规模：13 个源文件，约 4100 行（含 style.qss 样式表）

### 3.2 关键字段

**groups**：`{ "Agile": { "category": "Shared", "self_weight": null, "order": 1 } }`。`self_weight` 来自 Tree Multipliers，null=无；`category` ∈ Shared/Exclusive/Weapon/Armor/Fighting Style/Special/Always。

**backgrounds**：`{ "adventurous_noble_background": { "display_zh":"冒险贵族", "group_rolls":{...}, "multipliers":{...}, "melee_only":true, "exclusive":{...}, "base_probabilities":{...}, "icon":"..." } }`。`group_rolls` 是对 `default_group_rolls` 基线的 +/- 修正；`exclusive` 有 fixed/prob/none/mixed 四模式。

**traits**：`{ "Athletic": { "display_zh":"健壮", "weights":{ "Agile":2.0 }, "icon":"..." } }`。权重值用数字或字符串 `"+"`（强加成）、`"0"`（禁用）。

**exclusive_groups**：`{ "Knave": { "other_weights": { "Dagger":"+", "Light Armor":"+" } } }`——骰出某专属组后对后续组的级联修正。

### 3.4 关键语义约定

- 特性权重 `+`：强加成，建模为 `config.big_weight = 10.0`（**非 9999**，已调低避免概率饱和到 100%、保留区分度）。
- 特性权重 `0`：禁用该组（权重置 0，该组不出现）。
- `Melee Only`：背景限定纯近战，关闭 Bow/Crossbow/Throwing/Ranged 等远程组。
- 专属组 prob 模式：按概率骰其一（如 `Knave:30/Trapper:70`），影响后续 Weapon 等组权重。
- **默认骰组基线**（config.default_group_rolls）：Exclusive 0.5、Shared 2、Weapon 3、Armor 2、Fighting Style 2、Special 0。背景 group_rolls 是对此基线的 +/- 修正，引擎层叠加。
- **Tactician 特殊处理**（决策 A）：无概率矩阵列、基础概率 null，靠 self_weight(0.1) + multipliers + 特性 + 专属组级联合成。
- **专属组四模式**（决策 B）：None=不骰 / +=强加成 / prob=按概率 / mixed=强加成+概率混合。组名以概率矩阵 47 组为规范键。

## 四、核心算法（engine.py）

### 4.1 权重合成（正向）

对背景 B、特性集合 T、目标组 g，合成权重：

```
W(g) = base_prob(B,g) × self_weight(g)
     + background_multiplier(B,g)
     + Σ trait_weight(t,g)
     + attribute_star_weight(g)
     + projected_attr_weight(g)
     + Σ exclusive_cascade(E,g)
```

规则：任一来源使权重为 0 则该组不出现（概率记 0）；Melee Only 背景强制远程组权重为 0；同类别内各组按权重归一得到被骰中概率。

**按背景属性注入真实数据**（v0.3.0 新增）：`attribute_star_weight` 和 `projected_attr_weight` 不再使用全局默认中值。引擎新增 `_bg_expected_talent_stars(bg_id)` 从星级分布（60/30/10）加权计算期望约 1.5 星，以及 `_bg_projected_attrs(bg_id)` 从背景属性范围中值 + 星级推导 11 级投影属性。无属性数据的 4 个 mod 背景（如 `rf_old_swordmaster`）回退全局默认。手动传入 `talent_stars` 参数时可覆盖引擎推导值。

**保证专属组覆盖语义**（v0.2.0 新增）：mod 将专属组（Exclusive）的分配作为独立保证机制，在类别骰组逻辑**之外**执行。引擎通过 `_get_guaranteed_exclusive_probs()` 计算保证概率（fixed→100%、prob→按 picks 归一化概率、mixed→forced 组 100% + picks 组加权），并在 `forward_simulate()` 和 `appear_prob_analytic()` 中用保证概率**覆盖**（非叠加）Exclusive 类别的普通 0.5× 骰组逻辑。

**分数骰组语义**（v0.3.0 补充）：骰组数 n 含小数部分 f 时，解释为「有 f 的概率额外多骰一次」，解析式 `P = (1-f)·P(⌊n⌋) + f·P(⌊n⌋+1)`，MC 端每次采样按 f 概率多骰一次，两端口径一致。Exclusive 类别基线 0.5 由此获得真实语义——此前 `int(round(0.5))=0` 使**非保证但有正权重**的专属组恒为 0%。受影响 6 例：swordmaster/rf_old_swordmaster→Swordmaster 25%、fisherman/old_gladiator→Trapper 25%、barbarian→Wildling 10.2%（= 0.5 × 目标权重占比；**已保证组仍占池内权重**，如 swordmaster 的池为 {Soldier:1.0, Swordmaster:1.0} 故 0.5×0.5=25%）。保证组概率与整数骰组路径完全不变。若后续查证 mod 实际将已保证组排除出额外骰组池（那样 Swordmaster 应为 50%），在 `_category_pool` 调用处过滤即可。

### 4.2 单组出现概率

设类别 C 骰 n_C 次（来自背景 group_rolls + 基线，可含小数，见 §4.1 分数骰组），组 g 属于 C：

- **解析模式（analytic）**：用**无放回抽样解析近似**——`_appear_prob_no_replacement` 按位掩码递归「逐轮抽中移出 + 重新归一」，(mask, k) 子结果在单次调用内记忆化，顶层结果进实例级缓存（见 §4.6）。**非** `1-(1-p)^n` 有放回（后者在一组权重极大时严重低估其他组）。类别权重池由 `_category_pool` 按（背景, 特性, 分支）合成并缓存，同类别多目标共享。
- **蒙特卡洛模式**：`forward_simulate` 走**联合模拟** `_forward_monte_carlo_all`——每次采样完整生成一次角色（全类别骰组），同时统计**所有组**的出现次数，20000 次采样、固定种子可复现；权重按分支在采样循环外预计算（`_mc_branch_plans`）。单组 API `appear_prob_monte_carlo` 保留（同样吃预计算收益）。此前逐组独立模拟 + 循环内重复合成权重，单背景全组耗时 42s，重构后 0.12s（约 350×）。
- 两种模式结果一致（偏差在采样噪声内，30k 采样约 ≤0.007）。

### 4.3 反向推导

遍历全部背景 × 特性组合，计算目标组出现概率，按得分降序排列。

- **阈值化 Top-N 排名**（v0.2.0 重写）：收集所有得分>0 的 (背景, 特性组合) 行 → 按背景去重（同背景只保留得分最高的行，同分时取字典序最小的特性组合）→ 按得分降序排列 → 取前 top_n 个（默认 15，范围 5-50）。
- **与旧语义的区别**：旧版仅返回 `score == max_score` 的并列项，导致多目标时单一峰值独占全部结果（如 Shield+Vigorous 仅返回老剑圣 2 行）；新版返回一梯队候选背景，玩家看到排名而非唯一最优解。
- **纯度次级排序**：返回行数 > tiebreak_limit=20 时，计算纯度（非目标组概率总和，越小越纯粹），同分时纯度升序。v0.3.0 补充：纯度改为复用 `forward_simulate`（analytic 走 LRU 缓存、MC 走联合模拟），与旧的逐组独立计算口径逐位等值，但每行从 ~46 次独立计算降为 1 次（缓存命中近乎免费）。
- 返回结构含 `max_score`/`tied_count`（与最高分并列的行数）/`results`（去重后前 top_n 行）/`tiebreak_used`。
- **剪枝动态调整**：`prune_threshold` 自动取 max(20, top_n×2)，确保候选池足够返回 top_n 个不同背景。

### 4.4 合法性判定

若所选背景+特性导致所有可骰类别都无可用组（全被禁用或权重为 0），提示「无可用方案」并给原因（如「所选特性禁用了全部可骰组」）。

### 4.5 组树显示策略（v0.2.0 新增）

- **>0% 全显示**：`skill_tree_widget.py` 的 `set_trees()` 激活所有概率>0 的组（不再截断前 `DEFAULT_MAX_GROUPS=5` 个）。
- **保证专属组按实际概率显示**：任务 A 修复保证组概率后，Raider→100%、Soldier→100%、Knave→30% 等保证组自动以正确概率进入矩阵。
- 保留 chip 标签栏的增删交互（用户可手动移除不想看的组）。
- 保留 `DEFAULT_MAX_GROUPS` 常量定义（其他地方可能引用），但 `set_trees` 不再使用。

### 4.6 缓存体系与线程模型（v0.3.0 补充）

引擎持有五层实例级缓存，全部假定 GameData 静态只读：

| 缓存 | 键 | 上限与淘汰 | 说明 |
|---|---|---|---|
| `_forward_cache` | (bg, traits, 开关, stars) | 200，真 LRU（命中 move_to_end，淘汰最久未用） | forward_simulate analytic 结果 |
| `_norepl_cache` | (target, 权重签名, n) | 50000，超限整体清空 | 无放回递归**顶层**结果；(mask,k) 子结果只在单次调用内局部记忆化——其依赖 target 与权重签名，跨调用不可复用，实例级存储只会无界膨胀（曾积累 10211 条，局部化后 111 条） |
| `_pool_cache` | (bg, traits, cat, 分支, 开关, stars) | 20000，超限整体清空 | 类别权重池；同类别多目标共享一张池 |
| `_bg_stars_cache` | bg_id | 无上限（≤99 条） | 天赋星期望，纯静态推导 |
| `_bg_proj_cache` | bg_id | 无上限（≤99 条） | 11 级投影属性；仅缓存默认路径，显式传 talent_stars 时绕过 |

约定：
- **缓存返回的 dict 是共享对象，调用方不得修改**（`_bg_expected_talent_stars`/`_bg_projected_attrs`/`_category_pool` 均如此，现有调用方全部只读）。
- **引擎实例非线程安全**（dict/OrderedDict 复合操作无锁）。跨线程使用时各自实例化引擎、共享 GameData——`ReverseDeriveWorker` 即在构造时 `SkillEngine(engine.gd)` 建独立实例，避免推导期间用户切到正向页并发计算时的缓存竞争。
- 超限「整体清空」而非精细 LRU：单条重算成本低（毫秒内），精细淘汰的簿记开销不值。

**View 对齐**：`SkillTreeView` 必须设 `setAlignment(Qt.AlignLeft | Qt.AlignTop)`。`QGraphicsView` 默认把小于 viewport 的 scene 居中显示，会导致网格左、上出现空白（曾出现左侧与上方留白问题）；左上对齐后网格紧贴左上角，**右侧空白是故意保留的**——预留给「+ 添加组」按钮与未来新增列的扩展空间。垂直滚动条策略 `ScrollBarAlwaysOff`（不出现滚动条）。

> **行高取值权衡**：`CELL_H` 从 44 增至 56，是为给概率光环留出垂直空间——节点直径 40px 加光环外缘约 10px 后视觉高度约 50px，44px 行高会使相邻行光环互相挤压重叠，56px 留出约 6px 呼吸空间。副作用是 7 阶总高度随之增大，窗口过小时第 7 阶会被裁切（因垂直滚动条关闭）——这是预期行为，用户可自行拉大窗口以一屏完整显示 1–7 阶；如需在小窗口下也能看全，可临时把 `setVerticalScrollBarPolicy` 改为 `ScrollBarAsNeeded`，但这与「无滚动条一屏看全」的设计意图相悖。

API：`set_trees(trees)`（每项 dict 含 `group_id`/`group_name`/`probability`/`tiers`，tiers 是 `{"Tier 1":"技能名\n描述",...}` 或纯名字 `"技能名"`）、`set_lang(lang)`（切换 tooltip 语言并 `_rebuild`）、`clear()`、`scroll_to_group(group_id)`。`SkillNode.__init__` 签名含 `lang="zh"` 参数。

#### 5.3.1 ColumnHeaderOverlay 列头方案（v0.2.0）

**问题背景**：原列头绘制位于 `SkillTreeView.drawForeground()`，在 PySide6/Qt6 中遇到两个问题：(1) `drawForeground` 的 painter 被裁剪到 `exposedRect`，即便 `FullViewportUpdate` 也不可靠——向右滚动后初始视口外的列头被裁掉，且坐标随滚动偏移；(2) `mapFromScene` 返回整数 QPoint，存在亚像素截断导致列头与下方节点轻微不对齐。

**尝试方案**：
- 方案 A：调整 scrollbar 值、`mapFromScene`、`setClipping(False)` 组合——均无法解决 Qt6 的 painter clip region 覆盖问题。
- 方案 B1（初版）：独立 `ColumnHeaderOverlay` QWidget 浮层，使用 `mapFromScene` + `viewport().pos().x()` 计算列头 x 坐标。**解决了缺失/偏移主问题，但存在轻微 1-2px 不对齐。**
- 方案 B2：改用 `viewport().geometry()` 定位浮层 + `viewport().mapTo()` 坐标转换——**浮层初始化时 viewport 尚未完成布局，geometry 不可靠，导致列头完全消失。**
- 方案 B3：恢复浮层全视口宽度 + `viewportTransform()` 矩阵直接计算 x 坐标——**最终采用**。

**最终方案**：

1. **架构**：`ColumnHeaderOverlay` 是 `QWidget` 子控件，作为 `SkillTreeView` 的 child，位于 `(0, 0, view.width(), COL_HEADER_H)`。`setViewportMargins(0, COL_HEADER_H, 0, 0)` 在视口上方预留 COL_HEADER_H 空间。浮层绘制列头背景条、角落补丁（行头区域）、概率色块、列名文字和高亮动画。

2. **坐标计算**（关键）：使用 `viewportTransform()` 矩阵直接做 scene→viewport 映射，避免 `mapFromScene` 的整数截断：

```python
t = self._view.viewportTransform()
scene_x = float(MATRIX_LEFT + col * CELL_W)
vp_x = t.m11() * scene_x + t.m31()          # float 精度 scene→viewport
x = int(self._view.viewport().x() + vp_x)    # viewport→overlay 坐标
```

3. **滚动同步**：`_on_h_scroll` 同时调用 `self._header_overlay.update()` 和 `self.viewport().update()`，确保滚动时浮层与视口同步刷新。

4. **drawForeground 简化**：仅保留行头标签绘制（`drawForeground` 的 painter 在 Qt6 中对垂直方向裁剪可靠），列头全部移至浮层。

5. **Tooltip 改进**（同期修复）：
   - 中文 tooltip 始终显示英文 → `perk_zh._app_dir()` 增加 `sys._MEIPASS` 检查，PyInstaller 打包后正确读取 `perk_i18n.json`。
   - Tooltip 约 2 秒自动消失 → 用 `QLabel`（`Qt.ToolTip | Qt.FramelessWindowHint`）替换 `QToolTip.showText()`，`hoverEnterEvent` 显示、`hoverLeaveEvent` 隐藏，支持长时间停留。

**涉及文件**：`skill_tree_widget.py`（`ColumnHeaderOverlay` 类 ~70 行 + `SkillTreeView` 修改）、`perk_zh.py`（`_app_dir()` 修复）。

- **英文背景名规范化**（`GameData._en_display_name` 静态方法）：英文界面下背景名按规则规范化（见 §5.7），中文界面与引擎逻辑、id 不变。
- trait/group id 不含后缀，英文下直接返回 id 已是可读短语（如 Arena Fighter、Agile），无需规范化。
- **技能 tooltip 语言切换**（v0.0.3 补充）：`SkillTreeWidget` 持有 `self._lang`，`SkillNode` 持有 `self._lang`。`tab_forward`/`tab_builds` 的 `on_data_ready` 末尾调 `self.skill_tree.set_lang(gd.lang)`，若有 `_last_results` 则重新 `_fill_skill_tree` 刷新 tooltip 语言。切语言时主窗口调各页 `on_data_ready`，tooltip 实时跟随。英文描述来自 data.json perk_trees（xlsx 原文），中文描述来自 perk_zh.py 的 `PERK_DESC_ZH`（完整直译，150 个）。
- **perk_zh.py 翻译原则**：英文描述直接用 xlsx 表格原文（存于 data.json perk_trees，格式 `名字\n描述`）；中文 `PERK_DESC_ZH` 按英文原文**完整直译**，保留所有数值与机制细节（如 "50% increased Action Point cost" → "行动点消耗增加50%"）。术语统一：AP=行动点、Fatigue=疲劳、Resolve=决心、Initiative=主动值、Hitpoints=生命值、Melee/Ranged Skill=近/远程技能、Defense=防御、Reach=攻击距离、Hitchance=命中率、Armor=护甲、Bleeding=出血、Stun=晕眩、Daze=发昏、Stagger=踉跄、Rooted=定身、Netted=被网。

### 5.7 英文背景名规范化规则

`_en_display_name(bg_id)` 按顺序处理：

1. 去 `_background` 后缀
2. 去 `rf_` 前缀（孤立背景内部标记）
3. `_southern` → 提到词首作 `Southern ` 前缀（如 `assassin_southern` → `Southern Assassin`）
4. `_1h`/`_2h`/`_ranged` → `1H`/`2H`/`Ranged` 拼词尾（如 `companion_1h`→`Companion 1H`）
5. 其余下划线转空格 + 首字母大写，介词冠词集合 `{on,the,of,for,in,and,to,a,an}` 句中保持小写
6. 特殊修正：`Kings Guard` → `King's Guard`

99 背景全验无撞名。示例：`adventurous_noble_background`→`Adventurous Noble`、`killer_on_the_run_background`→`Killer on the Run`、`regent_in_absentia_background`→`Regent in Absentia`、`rf_old_swordmaster_background`→`Old Swordmaster`。

### 5.8 导航实现选型

曾尝试 QPushButton + QButtonGroup 的 `idClicked` 信号，但 PySide6 中信号连接存在**重复触发问题**导致页面不切换，已弃用。最终用 `QListWidget` + `currentRowChanged` 信号，`_on_nav(row)` 直接调 `stack.setCurrentIndex(row)`。选中态左侧 3px 金色竖线。

### 5.9 全局功能

- 导航：QListWidget 侧边栏，选中态金色竖线

## 六、路径处理与打包

### 6.1 PyInstaller 路径兼容（关键）

单文件 exe 运行时 PyInstaller 创建临时目录 `_MEIPASS`（只读），打包内置的 data.json 从那里读取，但 settings.json/builds/ 无法写入只读目录。方案：

- `data.json`：优先从 **exe 同级目录**读取（便于用户更新），找不到再用打包内置副本（`_MEIPASS`）。
- `settings.json` 和 `builds/`：始终写入 **exe 同级目录**（`exe_dir()`），**绝不写入 `_MEIPASS`**。
- `icon_provider.py` 和 `build_parser.py` 的目录定位统一用 `sys.executable`（frozen 时）。

`exe_dir()` 实现：frozen 时返回 `os.path.dirname(sys.executable)`，否则返回脚本所在目录。

### 6.2 build_safe.spec 关键配置

- 单文件模式（onefile），console=False（无控制台窗口）
- `datas=[('data.json','.'), ('perk_trees.json','.'), ('perk_i18n.json','.'), ('../VERSION','.')]`（打包内置作兜底；exe 同级副本优先）
- icons 不打包进 exe，放 exe 同级；源码库中在 `src/icons/`（入 Git），CI 打 Release 包时从仓库复制
- `hiddenimports` 显式声明：tab_forward/tab_reverse/tab_builds/tab_about/main_window/icon_provider/data_loader/engine/skill_tree_widget/perk_zh + PySide6.QtCore/QtGui/QtWidgets
- 排除 QtWebEngine 等大模块减体积，产物约 46MB

### 6.3 打包步骤（Windows）

前置条件：Python 3.10+，项目根目录 `pip install -r requirements-dev.txt`（含 PySide6 + PyInstaller；运行时依赖 requirements.txt 仅 PySide6）。

```cmd
cd "G:\AI PROJECT\BBR_SkillSim\src"
python -m PyInstaller build_safe.spec --noconfirm
```

产物在 `src\dist\BBR_SkillSimulator.exe`，将其与 `data.json`、`perk_trees.json`、`perk_i18n.json`、`icons\` 复制到发布目录。推送 `v*` 标签时 GitHub Actions（build.yml）自动完成同样流程：写 VERSION → 装 requirements-dev → **先跑测试** → PyInstaller → 汇集产物（图标复制失败会显式报错，不再 `|| true` 静默吞掉）。

> **用 `python -m PyInstaller` 而非 `pyinstaller`**：后者依赖 pyinstaller.exe 在 PATH，若 Python Scripts 目录未加 PATH 会报「不是内部或外部命令」。`python -m` 走 Python 模块路径绕开此问题。

### 6.4 build_simple.bat（一键打包）

GBK 编码 + CRLF 行尾（中文版 cmd 默认 GBK，UTF-8 bat 会乱码 + 命令错位）。脚本逻辑：`chcp 65001` → 检查 python 可用 → 检查 PyInstaller（缺失自动 pip install）→ 检查 build_safe.spec 存在 → `python -m PyInstaller build_safe.spec --noconfirm` → errorlevel 失败分支显示「[失败]」并退出（避免命令失败仍显示「完成」）。

> 必须在 src 目录下运行（脚本用相对路径 `build_safe.spec`，且有 spec 存在检查）。

### 6.5 运行时目录结构

```

  data.json                  ← 必需（exe 同级优先于内置兜底）
```

图标命名：`icons/<类别>/<英文规范键>.png`，背景如 `adventurous_noble_background.png`、特性如 `Athletic.png`、组如 `Agile.png`。缺失图标自动用占位图（带首字母彩色方块），不影响使用。

## 九、开发约定与已知工程坑

### 9.1 文件写入完整性（重要）

本开发环境（Claude 沙箱）的 Edit/Write 工具对长 Python 文件（engine.py、data_loader.py、main_window.py、tab_builds.py 等）**反复出现末尾截断 / 引入 null 字节**问题，且**截断后 `ast.parse` 仍可能通过**（因截断点语法凑巧完整）。

**应对**：
- **不能只靠 ast 校验**，必须 `tail -N` 目视确认末尾方法完整 + grep 关键方法名存在。
- **写长 Python 文件一律用 bash heredoc**（`cat > file << 'EOF'`），这是本环境唯一可靠方式。
- 发现截断时用 bash heredoc 重建 tail；可参照同项目同类文件（如 tab_builds.py 的 `_fill_skill_tree` 可对照 tab_forward.py 同名方法）。

历史踩坑：data_loader.py 丢了 `load_data` 尾 + `__main__`；main_window.py 丢了 `_retry` 尾 + `_build_statusbar`/`_on_nav`/`_setup_shortcuts`；tab_builds.py 丢了 `_fill_skill_tree` 的 trees.append 字典闭合；tab_reverse.py 的 `_fill_table` 曾截断在 `p_ite` 处（NameError）；**v0.0.3 补充又踩** skill_tree_widget.py（末尾 `_animate_scroll` 注释截断）、tab_forward.py（末尾 `_on_table_click` 截断在 `self.skill_tree.scroll`，应为 `scroll_to_group(gid)`，潜伏 bug 已补全）、开发文档.md（写入截断）。

**v0.0.3 补充新增经验**：
- **Edit 工具改长文件每次都截断末尾**（即使只改中间小块也会截断文件末尾）。**改用 Python 脚本** `open(encoding='utf-8').read()/replace()/write()` 方式修改长文件——经多次验证不截断（Edit 工具截断而 Python 脚本不截断，原因不明，可能 Edit 工具走不同的写入路径）。写后立即 `ast.parse` + `tail -N` 目视 + `data.count(b'\x00')` 检查 null 字节。
- **写入挂载点大文件也可能截断**（开发文档.md、data.json 都遇到过），写后必须校验字节完整性 + 末尾换行。
- **从 .pyc 反汇编恢复截断源码**：decompyle3 不支持 Python 3.10，改用 `dis` + `marshal` 反汇编字节码看方法完整逻辑——如确认 `_animate_scroll` 到 `anim.start()` + `POP_TOP` + `RETURN_VALUE` 就结束（被截断的只是后续注释，不影响功能）。`.pyc` 头部 `Size of source mod 2**32` 字段记录原源码字节数，可对比判断是否截断。
- **.gitignore 曾有 154 个 null 字节填充损坏**（末尾大片 `\x00`），用 `data.replace(b'\x00', b'')` 清理。

### 9.2 QCompleter Model

`QCompleter()` 无参构造时无内置 model，调用 `.model().setStringList()` 报 NoneType。需显式创建 `QStringListModel` 存为 `self._bg_model`，通过 `completer.setModel(self._bg_model)` 注入，再用保存的引用调 `setStringList()`。

### 9.3 Qt 控件无 len()

`len(QListWidget)` 报错，改用 `.count()`。

### 9.4 其他工程决策

- **导航**：QListWidget + currentRowChanged（QButtonGroup.idClicked 在 PySide6 重复触发，已弃用）。
- **数据加载后显式检查 None**：load_data 返回 None 会漏到 SkillEngine 导致 NoneType big_weight，需数据加载后检查。
- **`_toggle_lang` 开头加 `if self.gd is None: return`**：避免语言切换时 gd 为 None 报 NoneType set_lang。
- **ScrollArea 高度塌缩**：QFormLayout 内 ScrollArea 会塌缩，需移出 FormLayout + 加 minHeight + stretch。
- **列表推导式变量名 typo 潜伏 bug**：`_finish_derive` 中 `[c for ... for cb, _ in cbs if cb.isChecked()]` 表达式变量 `c` 未定义（应为 `cb`），但因方法前半段 `if not rd["results"]` 提前 return，单目标/无结果时不走到该行——bug 潜伏到多目标推导才暴露。教训：列表推导式涉及多层 for/解包时，表达式变量名容易与内层变量名脱节，写完后应检查表达式变量是否在内层被实际绑定。**v0.3.0 补充**：`retranslate` 中潜伏着完全同款的第二处 `c`→`cb`（带结果切语言必崩），由 ruff F821 静态检出——此类 bug 靠人眼复查不可靠，**ruff 进 CI 后在提交前即暴露**，这是把 lint 纳入门槛的直接理由。
- **循环变量遮蔽导入**（v0.3.0 补充）：`skill_tree_widget.py` 三处 `for t in ...` 遮蔽 i18n 的翻译函数 `t`——与 v0.2.0 `_on_add_menu` 崩溃同根源，当时只修了出崩溃的那一处。ruff F402 一次性检出全部残留（改名 `tree`）。教训：单字母通用名（t/l/c）与常用导入极易撞名，修 bug 时应全文件搜同模式而非只修报错点。
- **实例级缓存必须有界或局部化**（v0.3.0 补充）：`_norepl_cache` 曾把递归 (mask,k) 子结果连同顶层结果一起塞进实例字典且无上限——子结果依赖 target 与权重签名，跨调用**永远不会命中**，纯属内存泄漏（一次多特性反向可积累千万级条目）。教训：加缓存前先问「这个键跨调用真的会重复吗」；子结果记忆化用调用内局部 dict，实例级只存会复用的顶层结果并设上限。
- **蒙特卡洛先看循环不变量**（v0.3.0 补充）：旧 MC 在 20000 次采样循环内逐组重新合成权重（权重只依赖分支，与采样无关），且 forward 对 ~48 个组各自独立跑全量采样（一次联合模拟即可统计所有组）——两层冗余合计约 350×。教训：性能优化先找「循环内不随迭代变化的计算」和「同一份随机过程被重复模拟」。
- **引擎跨线程共享**（v0.3.0 补充）：反向 worker（QThread）曾与主线程共享同一引擎实例，缓存 dict/OrderedDict 复合操作无锁，推导期间切正向页并发计算有竞争风险。改为 worker 构造独立 `SkillEngine(gd)`（GameData 只读可共享），引擎实例化成本可忽略。
